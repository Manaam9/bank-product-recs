# 💳 Рекомендательная система банковских продуктов

## 📌 Описание проекта

Цель проекта — разработать рекомендательную систему, которая предсказывает, какой банковский продукт целесообразно предложить клиенту.

В качестве объектов рекомендации рассматриваются банковские продукты: счета, карты, кредиты, депозиты, инвестиционные и другие финансовые сервисы. Проект строится как production-oriented ML pipeline: от анализа данных и постановки задачи до логирования экспериментов, подготовки сервиса и мониторинга.

---

## 🎯 Цель проекта

Построить модель, которая на основе истории клиента и его текущего продуктового портфеля предсказывает, какие **новые банковские продукты** с наибольшей вероятностью будут востребованы в следующем периоде.

---

## 🧠 Постановка задачи

Задача формулируется как:

> предсказание новых продуктов клиента на основе его предыдущего состояния и истории взаимодействия с банком.

Особенности задачи:

* данные имеют выраженную временную структуру;
* у клиента может появиться несколько новых продуктов одновременно;
* целевая переменная разреженная и несбалансированная;
* задача ближе к **recommendation / ranking**, чем к классической бинарной классификации.

---

## 📊 Описание данных

Используется датасет с историей клиентов банка за 1.5 года.

Особенности данных:

* период наблюдений начинается с **28 января 2015 года**;
* данные представлены в виде **ежемесячных срезов**;
* каждая строка отражает состояние клиента в конкретный месяц;
* продуктовые признаки представлены бинарными колонками вида `ind_*_ult1`.

### Основные группы признаков

* временные признаки: `fecha_dato`, `fecha_alta`, `ult_fec_cli_1t`;
* идентификатор клиента: `ncodpers`;
* клиентские характеристики: `sexo`, `age`, `pais_residencia`, `segmento`;
* статусные и поведенческие признаки: `ind_empleado`, `ind_nuevo`, `antiguedad`, `indrel`, `tiprel_1mes`, `ind_actividad_cliente`;
* география и канал: `canal_entrada`, `cod_prov`, `nomprov`;
* финансовые признаки: `renta`;
* продуктовые признаки: `ind_ahor_fin_ult1`, `ind_tjcr_fin_ult1`, `ind_recibo_ult1` и др.

### Важное замечание

Текущие продуктовые признаки используются не как готовый таргет, а как основа для построения целевой переменной — **новых продуктов относительно предыдущего месяца**.

## 🔍 Этап 1. Исследование данных (EDA)

На первом этапе был выполнен первичный анализ данных в Jupyter Notebook.

### Что было сделано

* проведён обзор структуры датасета;
* проверены типы признаков;
* выполнена оптимизация памяти датафрейма;
* проанализированы пропуски;
* проверены дубликаты;
* проверена корректность категориальных значений;
* исследована временная структура данных;
* проанализирована длина истории клиентов;
* исследовано распределение количества продуктов на клиента;
* выполнен анализ популярности продуктов;
* сформирован и проанализирован таргет новых продуктов.

### Основные выводы EDA

* данные имеют корректную временную структуру: одна запись соответствует одному клиенту в конкретный месяц;
* полные дубликаты и дубликаты по ключу (`ncodpers`, `fecha_dato`) отсутствуют;
* часть признаков содержит значимые пропуски, при этом признаки `conyuemp` и `ult_fec_cli_1t` практически полностью пустые и исключаются;
* признак `renta` является важным, но требует обработки пропусков и выбросов;
* категориальные признаки в целом корректны, проблема unseen-категорий минимальна;
* большинство клиентов имеют достаточно длинную историю наблюдений, однако присутствует cold-start сегмент;
* большинство клиентов владеют 1–2 продуктами, что подтверждает потенциал кросс-продаж;
* популярность продуктов распределена неравномерно и имеет выраженный long-tail характер;
* таргет новых продуктов крайне разреженный: в большинстве наблюдений новые продукты отсутствуют.

### Итог EDA

По результатам анализа данные признаны пригодными для построения рекомендательной системы, однако требуют:

* обработки пропусков;
* очистки и унификации категориальных признаков;
* временного разбиения выборки;
* использования ranking-oriented метрик и моделей.

📁 Артефакт этапа: `notebooks/01_eda.ipynb`

## ⚙️ Этап 2. Подготовка инфраструктуры экспериментов

Для отслеживания экспериментов развёрнут MLflow.

### Конфигурация

* backend store: SQLite;
* artifact store: локальная директория;
* интерфейс доступен через MLflow UI;
* запуск выполняется отдельным shell-скриптом.

### Запуск

```bash
./scripts/run_mlflow.sh
```

### Доступ к интерфейсу

MLflow разворачивается на виртуальной машине и открывается через SSH-туннель:

```bash
ssh -i <ssh_key> -L 5000:localhost:5000 <user>@<vm_ip>
```

После этого интерфейс доступен локально по адресу:

```text
http://127.0.0.1:5000
```

### Назначение MLflow в проекте

MLflow будет использоваться для:

* логирования параметров моделей;
* сохранения метрик;
* хранения артефактов;
* сравнения запусков;
* фиксации воспроизводимых экспериментов.

📁 Артефакт этапа: `scripts/run_mlflow.sh`
